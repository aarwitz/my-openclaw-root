# Task Routing Rules

Use these rules when deciding who should handle a task.

## Rule 1: Clarify before routing
If the request is vague, underspecified, or missing constraints, Jerry should clarify or structure it first.

## Rule 2: Route by bottleneck
Assign based on the real bottleneck:
- technical execution bottleneck → Aaron
- autonomous cross-domain judgment bottleneck → Taylor
- coordination / structure / research bottleneck → Jerry

## Rule 3: Match communication style
- Aaron: concise, exact, concrete, low-fluff
- Taylor: concise, structured, reasoning-rich, non-patronizing
- Jerry: adaptive, but should reduce ambiguity and coordination load

## Rule 4: Respect autonomy
Do not over-scaffold tasks in a person’s area of expertise.

## Rule 5: Use explicit ownership
For shared tasks, define:
- owner
- reviewer
- support
- expected deliverable
- success criteria

## Rule 6: Promote stable patterns
Log new observations in daily memory first. Promote only durable patterns into `people/*.md`.

## Example owner/reviewer/support patterns
- Aaron owner / Taylor reviewer / Jerry support
- Taylor owner / Aaron reviewer / Jerry support
- Jerry owner / Aaron approver / Taylor reviewer
